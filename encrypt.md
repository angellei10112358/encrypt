# 文本加解密工具 — 项目文档

## 1. 项目概述

一个纯前端的网页工具，提供基于 passphrase 的文本加密与解密功能。

**使用流程：**
1. Alice 在网页上输入明文和 passphrase，点击加密，得到密文字符串。
2. Alice 将密文和 passphrase（建议通过不同渠道）交给 Bob。
3. Bob 在同一网页上输入密文和 passphrase，点击解密，还原出原始明文。

**核心原则：**
- 纯前端实现，**不需要任何后端**。所有加解密计算通过浏览器原生 Web Crypto API（`crypto.subtle`）在用户本地完成。
- 明文和 passphrase 永远不离开用户设备，不发送到任何服务器。
- 最终产物为**单个自包含的静态 HTML 文件**（HTML/CSS/JS 全部内联），用户可下载后离线使用，便于审计。

## 2. 技术架构

- 单文件静态页面：`index.html`，内联所有 CSS 和 JavaScript。
- **禁止引入任何第三方 JS 库**（包括 crypto-js 等），以消除供应链风险。仅使用浏览器原生 API。
  - 特别说明：不得使用 crypto-js 的默认 CBC 模式和弱 KDF（`EVP_BytesToKey`），该方案缺乏完整性校验且抗暴力破解能力差。
- 无任何网络请求：无统计埋点、无外部字体、无 CDN 资源。如果未来确需引入 CDN 资源，必须添加 SRI 校验。

## 3. 加密方案（必须严格遵守，不得自行发明加密逻辑）

### 3.1 算法

| 环节 | 方案 |
|------|------|
| 密钥派生 | PBKDF2，SHA-256，迭代次数 ≥ 600,000 |
| 加密算法 | AES-256-GCM（认证加密） |
| 随机数生成 | `crypto.getRandomValues()` |
| salt | 每次加密随机生成，16 字节 |
| IV | 每次加密随机生成，12 字节（GCM 标准长度） |
| 文本编码 | 统一使用 UTF-8（`TextEncoder` / `TextDecoder`），确保中文等多字节字符跨设备一致 |

### 3.2 密文输出格式

拼接后整体 Base64 编码：

[版本号 (1 字节)] + [salt (16 字节)] + [IV (12 字节)] + [AES-GCM 密文]

- 版本号从 `0x01` 开始。目的是将来升级算法（例如换用 Argon2）时仍能识别并解密旧格式密文。
- salt 和 IV 不是秘密，随密文一起公开传输是安全的，但**每次加密必须重新随机生成，严禁复用或硬编码**。

### 3.3 解密行为

- 解析版本号，按对应版本的参数进行 PBKDF2 派生和 AES-GCM 解密。
- GCM 解密失败（passphrase 错误或密文被篡改/损坏）时会抛出异常。此时向用户显示统一提示：**"密码错误或密文已损坏"**。不要也无法区分具体原因。

## 4. 安全要求

### 4.1 部署

- **必须通过 HTTPS 部署**。原因：
  - 防止页面 JS 在传输中被篡改（如植入窃取 passphrase 的代码）；
  - `crypto.subtle` 和 `navigator.clipboard` 在非安全上下文（非 HTTPS 且非 localhost）下不可用。
- 配置合理的 CSP 响应头（如通过 `<meta http-equiv="Content-Security-Policy">`），禁止外部脚本和内联事件处理器之外不必要的来源。

### 4.2 防 XSS

- 显示解密结果时必须使用 `textContent`，**严禁使用 `innerHTML`** 或任何将用户输入当作 HTML 解析的方式。
- 所有用户输入（明文、密文、passphrase）一律视为不可信数据。

### 4.3 不留痕

- passphrase 和明文不得出现在：URL（含 hash 和 query）、`localStorage`、`sessionStorage`、cookie、任何日志或网络请求中。
- passphrase 输入框使用 `type="password"`，并设置 `autocomplete="off"`。
- 不做任何形式的数据统计或上报。

### 4.4 passphrase 强度（不强制，仅引导）

- **不强制要求 passphrase 长度或复杂度**（避免影响用户体验）。
- 但需提供一个**非阻断式的口令强度提示条**，实时显示强度评估（弱/中/强），并在界面上给出建议文案，例如"建议使用 4 个以上随机词组成的短语"。
- 高迭代次数的 PBKDF2 是抵御离线暴力破解的主要技术手段。
- 页面上需包含使用说明，明确提醒用户：**密文和 passphrase 应通过不同渠道传递**（例如密文走邮件、密码走电话），否则加密失去意义。

### 4.5 信任模型局限（写入页面说明或 README）

- 即使以上全部做对，若托管服务器被入侵，攻击者仍可替换页面代码。缓解措施：代码开源、提供可下载的单文件离线版本供用户本地使用。

## 5. 界面与交互要求

### 5.1 整体布局

- **加密**和**解密**做成两个明确的 Tab 或模式切换，避免用户混淆输入框用途。
- 加密模式：明文输入框 + passphrase 输入框 + 加密按钮 + 密文输出区。
- 解密模式：密文输入框 + passphrase 输入框 + 解密按钮 + 明文输出区。

### 5.2 移动端兼容（必须同时适配 PC 和手机）

- 使用 viewport meta 标签 + 弹性/响应式布局。
- 所有输入框字号 ≥ 16px，避免 iOS Safari 聚焦输入框时自动缩放页面。
- 文本框（textarea）要足够大，方便在手机上粘贴长密文。
- 密文输出区提供**一键复制按钮**（`navigator.clipboard.writeText`，需 HTTPS），复制成功给出视觉反馈。
- 按钮和可点击区域尺寸适合触屏操作。

### 5.3 性能与反馈

- PBKDF2 高迭代次数在低端手机上可能耗时 1–2 秒：加解密进行中必须显示 **loading 状态**（按钮禁用 + 加载指示），避免用户误以为卡死。
- 加解密操作使用 `async/await`，不阻塞 UI。

### 5.4 错误提示

- 解密失败：统一提示"密码错误或密文已损坏"。
- 密文格式非法（Base64 解析失败、长度不足、版本号未知）：提示"密文格式无效"。
- 输入为空时给出友好提示，不执行操作。

## 6. 明确禁止事项（Agent 检查清单）

- ❌ 添加任何后端、API 调用或网络请求
- ❌ 自行设计/发明加密逻辑，或使用非认证加密模式（如 CBC 无 MAC）
- ❌ 复用或硬编码 salt / IV
- ❌ 使用 `innerHTML` 渲染用户相关内容
- ❌ 将 passphrase 或明文写入 URL / 存储 / 日志
- ❌ 引入第三方 JS 库
- ❌ 强制限制 passphrase 长度或复杂度（只做提示引导）
- ❌ 使用 `Math.random()` 生成任何加密相关随机数

## 7. 验收标准

1. Alice 在 PC 端加密的内容，Bob 在手机端用相同 passphrase 能正确解密（含中文、emoji、多行文本）。
2. passphrase 错一个字符即解密失败，且提示统一。
3. 同一明文 + 同一 passphrase 两次加密产生不同密文（salt/IV 随机性验证）。
4. 断网状态下页面功能完全正常。
5. 浏览器开发者工具 Network 面板中无任何网络请求（页面加载本身除外）。
6. 移动端（iOS Safari / Android Chrome）布局正常、无聚焦缩放问题、复制功能可用。
